5. Постмортем PR #18 (2026-04-30): max-out Sentry → 502

  • Что упало: один sentry_sdk.init с шестью integrations за раз. Какая-то сломалась на import → app не стартовал в контейнере → Caddy 502 → smoke упал → deploy job failed, но контейнер уже был recreated и сломан.
  • Почему CI не поймал: в pytest PV_SENTRY_DSN был пустой → _init_sentry возвращался ранним return → broken-import-путь никогда не выполнялся.
  • Как восстановили: gh run rerun <id-предыдущего-успешного> → старый sha образа задеплоился → /healthz 200. Затем revert PR #18 в main (PR #19), чтобы следующий любой push не воспроизвёл.
  • Что добавили чтобы не повторилось: PR #20 — tests/test_app_smoke.py заставляет _init_sentry пройти end-to-end с фейковым DSN и нулевыми sample-rates. Любой broken integration теперь падает в CI до того как PR смержится.
  • Что ещё хочется (в backlog): staging-окружение (одна и та же compose, отдельный suffix-домен), куда деплой идёт первым, а в прод — только если staging зелёный N минут.

5b. Постмортем 2026-05-04: dev/prod env divergence (PR #23-#25)

  • Что упало: PR #23 (HttpxIntegration), затем PR #24 (AsyncioIntegration), затем PR #25 (LoggingIntegration + attach_stacktrace + profiles_sample_rate) — каждый зелёный в CI, каждый 502 на проде. Cumulative: прод лежал с момента мерджа PR #23 до rollback'а на PR #22.
  • Корневая причина: sentry_sdk.integrations.httpx импортирует пакет httpx на module-level и кидает DidNotEnable("httpx is not installed") если пакета нет. У нас httpx>=0.27 сидел только в [project.optional-dependencies] dev. CI ставит pip install -e .[dev] → httpx есть → импорт работает → smoke зелёный. Прод-Dockerfile ставит pip install . (без [dev]) → httpx нет → импорт падает → uvicorn не может загрузить pv_api.main:app → workers умирают → контейнер не healthy → Caddy 502.
  • Почему PR #20 smoke не поймал: он гонится pytest'ом, а pytest есть только в [dev] — значит test environment всегда имеет [dev] packages. Тот же blind spot.
  • Метод диагностики, занявший 30 секунд: на VPS запустить упавший образ изолированно, не трогая прод-контейнер: docker run -d --name pv-api-debug --network pv_default --env-file /etc/pv/pv-api.env ghcr.io/.../pv-api:<bad-sha>, затем docker logs pv-api-debug --tail=120. Получил traceback с конкретной строкой.
  • Восстановление: gh run rerun на PR #22 build-deploy run → прод вернулся на a883e95. Затем PR #26 — revert трёх плохих в main. Затем PR #27 — fix-forward: httpx>=0.27 в core dependencies, все интеграции восстановлены, добавлен новый CI-job prod-deps-smoke (pip install . без [dev] + python -c "from pv_api.main import create_app; create_app()" с фейковым DSN).
  • Generalised lesson: dev/prod env divergence — это класс ошибок, не разовый случай. Любой from <pkg> import * в коде рантайма, тянущий новую зависимость, должен проверяться установкой prod-only deps в CI. Иначе CI зелёный, прод красный.
  • Watcher-предупреждение: gh run watch --exit-status в нашем auto-merge скрипте раньше выдавал «deploy: success» на PR #23/#24 — потому что контейнер ненадолго становился healthy в окне между deploy-step и smoke-step. Финальная истина — в gh run view <id> --json conclusion -q .conclusion.

6. Предписания себе после паузы / для следующей сессии

Чтобы при возврате не тратить полчаса на восстановление контекста. Эта же информация — в HANDOVER.md в репо.

  • Жив ли прод? Первая команда — curl https://api.pvlmtd.com/healthz. Если 200 + JSON — всё работает. Если нет — gh run list --workflow build-deploy.yml --limit 1, потом ssh + docker compose ... logs --tail=50 pv-api.
  • Бэкапы. Cron в 03:00 UTC. rclone ls vultr:pv-backups/ и ls -lh /var/backups/pv/ — там должна быть свежая дата.
  • Что делать первым: если pv-shared v0.1.0 готов — Order модели в pv-api + миграция + httpx в pv-console. Если ещё нет — backlog: один integration Sentry за PR / Grafana credentials / staging env.
  • Грабли клиента (повторяются каждую сессию): PS heredoc добавляет \r — оборачивать в ssh ... "tr -d '\r' | bash -s". Auto-linker рендерит *.com/.yml/.sh как markdown — не верить отображению, проверять cat -A; multi-line скрипты передавать через base64 -d. Из Vultr UI копировать только конкретное поле, не весь s3cfg-блок (Secret уже четыре раза утекал в чат).
  • SSH рвётся — в ~/.ssh/config на Windows блок Host pv-api-prod с ServerAliveInterval 30.
  • Чего НЕ делать без явного запроса: push в main любого репо (всё равно отобьёт branch protection); трогать pv-shared (зона PRC); менять /etc/pv/*.env на VPS без бэкапа предыдущей версии; docker compose down -v где-либо рядом с прод-volume'ами.
  • Если контекст-окно подходит к концу: сказать «закругляемся», обновить memory, открыть новую сессию и сказать «продолжаем pv-api» — memory подхватится автоматически. HANDOVER.md в репо — для случая когда memory недоступна (другой dev / другая Claude).
